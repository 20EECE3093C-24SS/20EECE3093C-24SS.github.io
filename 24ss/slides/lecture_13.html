Introduction to Design Patterns

Abstractions in Software Engineering

In software engineering, solutions to common problems are categorized into different levels of abstraction: architectural patterns, design patterns, and idioms.

  • Architectural Patterns: High-level strategies that concern the overall layout of software applications. They address fundamental issues like the structural organization and the flow of control.
  • Architecture: what components are needed, and what are their connections
  • Design: how the components are developed

This slide explains the relationship between architectural patterns, design patterns, and idioms. Architectural patterns are the most abstract, offering a blueprint for the overall structure and behavior of software applications. They help in defining the system’s high-level structure, ensuring it meets the required business and technical requirements. Design patterns are more specific, dealing with recurring design problems within a given context. They offer templates for solving common design issues, improving code readability, and making the software easier to maintain and modify. Idioms are at the bottom of this hierarchy, being highly specific to a programming language and focusing on small-scale problems, such as the use of particular syntax or language features to solve specific problems efficiently. Understanding the distinctions and relationships among these concepts is crucial for software engineers as they design and build complex systems.

Are Idioms Abstract?

Idioms are not abstract. They are specific, concrete patterns tailored to the intricacies of a particular programming language, offering direct guidance on effectively utilizing that language’s features.

  • Specific Guidance: Idioms detail the most effective ways to achieve certain tasks within a language, focusing on syntax and feature utilization.
  • Language-Specificity: Unique to each programming language, idioms address the nuances and specific capabilities of that language.

Note

Rather than being abstract, idioms are practical, providing clear, actionable advice for coding efficiently and clearly.

When addressing whether idioms are abstract, it’s important to clarify that they are, in fact, the opposite. Idioms represent concrete, practical knowledge and techniques specific to a programming language. They encapsulate the best practices for using a language’s unique features and syntax to solve common programming challenges effectively. This specificity makes them invaluable for developers seeking to optimize their code for readability, performance, and maintainability. Unlike more abstract concepts like design patterns, which provide general solutions to common design issues, idioms offer a straightforward, “how-to” approach to coding challenges, grounded in the practicalities of specific programming languages.

Patterns vs. Algorithms

Patterns and algorithms are both solutions to known problems, but they serve different purposes and are applied in different contexts.

  • Pattern: A high-level description of a solution to a common problem. Patterns provide a template rather than explicit instructions.
  • Algorithm: A clear, step-by-step set of actions to achieve a goal. Algorithms are precise and unambiguous.

Patterns are adaptable and can be implemented in various ways across different programs, leading to unique code instances. In contrast, algorithms are usually applied uniformly.

Patterns are often confused with algorithms, because both concepts describe typical solutions to some known problems. But they are different in nature and serve different purposes.

While algorithms provide a specific set of steps to solve a problem, patterns offer a general approach that needs to be adapted to the context of a particular project.

The key takeaway is that while an algorithm is a detailed guide, a pattern is more like a principle or a strategy that can be interpreted and applied in various ways depending on the specific requirements and constraints of the project. This flexibility makes patterns incredibly valuable in software design and architecture, allowing developers to solve problems in creative and adaptable ways.

Example: Python Algorithm

def linear_search(items, target):
    """
    Performs a linear search for a target item in a list.

    Parameters:
    - items: list of elements to search through.
    - target: the item to search for.

    Returns:
    - The index of the target if found, otherwise -1.
    """
    for index, item in enumerate(items):
        if item == target:
            return index  # Target found, return its index.
    return -1  # Target not found, return -1.

# Example usage:
items = [5, 3, 7, 1, 9, 2]
target = 7

result = linear_search(items, target)
if result != -1:
    print(f"Item found at index: {result}")
else:
    print("Item not found.")

Origin of Design Patterns

The concept of design patterns in software engineering was popularized by the book “Design Patterns: Elements of Reusable Object-Oriented Software,” published in 1994 by Erich Gamma, Richard Helm, Ralph Johnson, and John Vlissides. This book is commonly referred to as the “Gang of Four” (GoF) book, and it is considered the seminal work on design patterns.

The GoF book describes 23 design patterns, which are divided into three categories: creational, structural, and behavioral patterns. These patterns are intended to help developers solve recurring design problems and improve the flexibility and maintainability of their software.

Anatomy of Design Patterns

Design patterns are articulated through a structured format that encapsulates their essence and application. This format includes several crucial elements:

  • Intent: Defines the core objective of the pattern, outlining the specific design problem or issue it aims to resolve.
  • Motivation: Provides a deeper insight into the context, detailing why the pattern is necessary and how it addresses the identified problem effectively.
  • Structure: Illustrates the pattern’s composition and the relationships between its components, commonly represented through UML (Unified Modeling Language) diagrams for clarity and precision.

These elements ensure that design patterns are comprehensively understood, enabling developers to apply them effectively across different projects.

This slide elaborates on the structured components that constitute design patterns, emphasizing their importance in software development. Each element, from Intent to Structure, plays a pivotal role in conveying the pattern’s purpose, applicability, and implementation strategy. The Intent clarifies what the pattern is designed to do, serving as a quick reference to understand its goals. The Motivation section delves into the rationale behind the pattern, offering insights into the specific scenarios where the pattern proves beneficial. Lastly, the Structure, often visualized through UML diagrams, breaks down the pattern’s architecture, providing a visual roadmap of its implementation. This structured approach to describing design patterns not only aids in their understanding but also facilitates their practical application, ensuring developers can adapt them to solve specific design challenges effectively.

Classification of Design Patterns

Design patterns are categorized based on the nature of the problem they solve. The three primary categories of design patterns are:

  • Creational Patterns: Focus on object creation mechanisms, optimizing the process to increase flexibility and promote the reuse of existing code. Examples include Singleton, Factory Method, and Builder patterns, each facilitating a different approach to object instantiation.
  • Structural Patterns: Aim to assemble objects and classes into larger structures while maintaining flexibility and efficiency. These patterns ensure that changes to the system’s structure require minimal modifications. Examples include Adapter, Composite, and Proxy patterns, which help to build complex yet efficient systems.
  • Behavioral Patterns: Address effective communication and the distribution of responsibilities among objects. They enhance collaboration between objects, ensuring that they work together harmoniously. Examples include Observer, Strategy, and Command patterns, each fostering a different aspect of object interaction and cooperation.

Understanding these categories helps developers choose the right design pattern to address specific design challenges within their projects.

By understanding these classifications, software engineers can more effectively apply design patterns to solve common design problems, leading to more robust, maintainable, and flexible software architectures.

Design Principles

What are they?

Design principles in software engineering are fundamental guidelines or best practices that are followed to create high-quality, maintainable, and scalable software systems. These principles help software engineers to make decisions that ensure their software is robust, efficient, and easy to understand and modify.

Key Design Principles

  • KISS principle (keep it simple, stupid)

  • YAGNI principle (you ain’t gonna need it)

  • DRY principle (don’t repeat yourself; use abstractions, inheritance)

  • SOLID principle, a combination of design principles, and from where It’s mnemonic acronym has been taken.

    • Single responsibility (focus on on doing one thing well – high cohesion)
    • Open/closed principle (open for extension, closed for modification)
    • Liskov (behavioral) substitution principle (stronger specification; user of base class can use instance of derived)
    • Interface segregation principle (don’t force client to implement an interface if they don’t need it)
    • Dependency Inversion (high cohension, loose coupling principle)

Keep it simple, stupid (KISS)

The KISS principle (Keep It Simple, Stupid) advises against overcomplicating your code. The essence of KISS is to seek straightforward solutions, making your code easier to work with, now and in the future.

Why KISS Matters

  • Quick Problem-Solving: Simple solutions mean faster development and fewer headaches when tackling challenges.
  • Ease of Maintenance: A straightforward codebase is easier to understand, maintain, and extend.
  • Adaptability: Simple code is more flexible, making it easier to modify or refactor as your project evolves.
  • Efficient Solutions: Achieve more with less. Simplifying your approach often leads to solving complex problems with fewer lines of code.
  • Quality Output: Ultimately, simplicity leads to higher-quality code—code that’s robust, reliable, and ready for real-world use.

By prioritizing simplicity with the KISS principle, you not only make your life as a developer easier but also enhance the overall quality and longevity of your software projects.

You ain’t gonna need it (YAGNI)

The YAGNI principle (You Ain’t Gonna Need It) champions a minimalist approach to software development: only implement features when they’re absolutely necessary. By resisting the temptation to add speculative functionality, you keep your codebase clean, straightforward, and manageable.

Why YAGNI Matters

  • Cut Complexity: Streamline your project by avoiding unnecessary features, making your code easier to understand, maintain, and extend.
  • Prevent Over-Engineering: Focus on what’s needed now, rather than what might be needed in the future, to prevent complicating your project with unused features.
  • Enhance Efficiency: Spend your time and resources wisely by working on features that deliver real value to your users now.
  • Improve Flexibility: A leaner codebase is easier to adapt and evolve as new requirements emerge, ensuring your project can grow and change without being weighed down by unused code.

By adhering to the YAGNI principle, you ensure your development efforts are efficient, your codebase remains clean, and your project stays aligned with user needs and priorities.

Single Responsibility Principle (SRP)

The Single Responsibility Principle advises that a class should have only one reason to change. This means each class should focus on a single job or functionality.

Why SRP Matters

  • Focus: Keeps classes focused and cohesive, making them easier to understand and manage.
  • Maintainability: Simplifies maintenance by isolating changes to a single part of the codebase.
  • Testability: Focused classes are easier to test, leading to more reliable software.

SRP promotes a modular approach to software development, enhancing flexibility and reducing the complexity of code.

Open/Closed Principle (OCP)

The Open/Closed Principle states that software entities (classes, modules, functions) should be open for extension but closed for modification. This encourages enhancing functionality without altering existing code.

Benefits of OCP

  • Enhanced Flexibility: Enables the addition of new features without changing existing code.
  • Reduced Risk: Minimizes the risk of introducing bugs in existing functionality.
  • Scalability: Facilitates the development of scalable systems that can grow with minimal changes to the core.

OCP advocates for a more sustainable approach to software development, promoting adaptability and robustness.

OCP is a fundamental principle that guides developers in creating software that is easy to extend without necessitating changes to existing code. This principle is vital for maintaining the integrity of a system while accommodating growth and change. By adhering to OCP, developers can add new features or behaviors through extension mechanisms, such as inheritance or composition, thereby avoiding the potential pitfalls associated with modifying existing code. This approach leads to more stable, scalable, and maintainable software systems.

Liskov Substitution Principle (LSP)

The Liskov Substitution Principle requires that objects of a superclass shall be replaceable with objects of a subclass without affecting the correctness of the program.

LSP Key Points

  • Consistency: Ensures that subclasses can stand in for their parent classes without disrupting the application.
  • Reliability: Increases the reliability of the code by maintaining behavior predictability across hierarchies.
  • Reusability: Facilitates the reuse of code, making software development more efficient.

LSP promotes the design of robust and interoperable object hierarchies that enhance the flexibility and reliability of software.

LSP focuses on the importance of ensuring that subclasses can be used interchangeably with their parent classes without altering the program’s functionality. This principle is crucial for the development of software that is flexible and easy to maintain. By ensuring that derived classes maintain the behavior expected by the interfaces they implement or the classes they extend, LSP helps in creating a codebase that is more modular and reusable, with components that can be easily substituted as needed.

Interface Segregation Principle (ISP)

The Interface Segregation Principle advocates for creating specific interfaces rather than one general-purpose interface. Clients should not be forced to depend on interfaces they do not use.

Why ISP Is Important

  • Decoupling: Promotes loose coupling by ensuring classes only depend on the methods they use.
  • Clarity: Enhances clarity and organization of code by separating interfaces into discrete sets of responsibilities.
  • Flexibility: Increases system flexibility and adaptability to change by facilitating more targeted implementations.

ISP encourages a more modular and cohesive code structure, improving maintainability and scalability.

ISP emphasizes the need for segregating interfaces into smaller, more specific ones so that clients only need to know about the methods that are relevant to them. This principle helps in reducing the complexity of code dependencies, making the system easier to understand and modify. By applying ISP, developers can create more modular software architectures, where changes in one part of the system have minimal impact on others, thereby enhancing the overall maintainability and flexibility of the codebase.

Dependency Inversion (DI)

The Dependency Inversion Principle involves two key points: High-level modules should not depend on low-level modules. Both should depend on abstractions. Additionally, abstractions should not depend on details. Details should depend on abstractions.

DIP Benefits

  • Decoupling: Reduces dependencies between high-level and low-level modules, making the system more modular.
  • Flexibility: Makes it easier to refactor, change, and test the software, as the core logic is not bound to specific implementations.
  • Scalability: Enhances the ability to scale and evolve the software over time by relying on abstract contracts rather than concrete implementations.

DIP advocates for a design that prioritizes abstractions, ensuring a more flexible and maintainable software architecture.

DIP is a crucial principle that guides the structuring of software components to reduce tight coupling and enhance modularity. By insisting that both high-level and low-level modules depend on abstractions rather than concrete implementations, DIP facilitates the development of systems that are easier to modify, test, and extend. This principle supports the creation of a codebase where changes in specific implementations do not necessitate widespread modifications, thus significantly improving the maintainability and adaptability of the software.

Abstract vs. Concrete Classes in Python

Abstract vs. Concrete Classes in Python

In Python, classes are either abstract or concrete, serving different roles in the development of object-oriented programs.

  • Abstract Classes:
    • Serve as templates for other classes.
    • Contain one or more abstract methods.
    • Cannot be instantiated directly.
    • Use ABC module and @abstractmethod decorator.
  • Concrete Classes:
    • Implement all methods and properties.
    • Can be instantiated to create objects.
    • Provide specific, executable code for their methods.

Abstract classes define a set of methods that must be created within any child classes built from the abstract class, while concrete classes are fully implemented classes that can be directly used to create objects.

This slide introduces the distinction between abstract and concrete classes in Python, laying the foundation for understanding their roles and uses in object-oriented programming. Abstract classes act as blueprints for other classes, specifying a set of methods that derived classes must implement, thereby enforcing a contract. In contrast, concrete classes are fully fleshed out and can be directly instantiated, making them ready for use in creating objects with specific functionalities. This distinction is crucial for designing well-structured and robust software systems.

Implementing Abstract Classes in Python

Abstract classes in Python are created using the ABC module and the @abstractmethod decorator to define abstract methods.

from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def calculate_area(self):
        pass

This Shape class cannot be instantiated and requires subclasses to provide an implementation for the calculate_area method.

Here, we demonstrate how to define an abstract class in Python using the ABC module and the @abstractmethod decorator. The Shape class serves as an abstract base class, defining a contract for its subclasses by requiring them to implement the calculate_area method. This pattern is fundamental in creating a consistent interface for all shapes, ensuring that each shape subclass can calculate its area but allowing for different implementations.

Concrete Class Implementation

A concrete class implements all abstract methods defined by its abstract superclass, making it instantiable.

class Circle(Shape):
    def __init__(self, radius):
        self.radius = radius

    def calculate_area(self):
        return 3.14159 * self.radius**2

The Circle class provides a specific implementation of the calculate_area method, making it a concrete class based on the Shape abstract class.

The Circle class illustrates a concrete implementation of the abstract Shape class. By providing a specific implementation for the calculate_area method, Circle becomes a concrete class that can be instantiated. This example shows how abstract classes serve as templates for concrete classes, guiding the implementation of necessary functionalities while allowing for the flexibility to handle the specifics of each subclass, such as calculating the area of a circle in this case.

Utilizing Concrete Classes

Once a concrete class is defined, it can be instantiated and used to perform its specific functions.

circle = Circle(5)
print(circle.calculate_area())

This code snippet demonstrates creating a Circle object and using it to calculate its area, showcasing the practical use of concrete classes.

This final part brings the concepts of abstract and concrete classes into a practical scenario. By instantiating the Circle concrete class and calling its calculate_area method, we see the direct application of these object-oriented principles. This example underscores the importance of concrete classes in the realization of abstract class templates, enabling the development of functional, object-oriented software systems.

Extending Abstract Classes

Concrete classes can extend abstract classes by providing implementations for all of the abstract methods defined in the abstract class. This allows concrete classes to utilize the structure and contracts defined by the abstract class while specifying their own behaviors.

Abstract Class Definition

from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def calculate_area(self):
        pass

    @abstractmethod
    def calculate_perimeter(self):
        pass

The Shape abstract class now includes a second abstract method, calculate_perimeter, requiring all concrete subclasses to implement it as well.

Concrete Class Implementation

class Rectangle(Shape):
    def __init__(self, width, height):
        self.width = width
        self.height = height

    def calculate_area(self):
        return self.width * self.height

    def calculate_perimeter(self):
        return 2 * (self.width + self.height)

The Rectangle class extends the Shape abstract class by implementing both calculate_area and calculate_perimeter methods, adhering to the contract established by the abstract class and providing specific functionality.

Utilizing the Concrete Class

rectangle = Rectangle(10, 5)
print(f"Area: {rectangle.calculate_area()}, Perimeter: {rectangle.calculate_perimeter()}")

This demonstrates creating a Rectangle object, a concrete implementation of the Shape abstract class, and using it to calculate both the area and the perimeter.

In this extended example, we delve deeper into how concrete classes inherit from and extend abstract classes in Python. By adding a new abstract method, calculate_perimeter, to the Shape class, we illustrate the requirements for concrete subclasses to provide specific implementations for all abstract methods defined in the abstract class. The Rectangle class exemplifies a concrete class that fulfills the contract of the Shape abstract class by implementing both required methods, showcasing the flexibility and power of abstract classes in enforcing a consistent interface while allowing for varied implementations among concrete subclasses. This approach is fundamental in object-oriented design, promoting both code reuse and adherence to defined contracts, thereby enhancing the maintainability and scalability of software systems.
