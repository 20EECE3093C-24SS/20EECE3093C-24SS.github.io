Introduction to Build Systems

What are programs?

Programs are essentially a series of commands that tell a computer what actions to perform. At their core, these commands are interpreted by the computer as binary or “machine code,” which varies across different types of processors, like x86-64 and aarch64 (ARMv8), each having their unique way of understanding instructions. For practical programming, we use high-level languages, which can be broadly categorized into compiled and interpreted.

  • Compiled languages, such as C, C++, and Java, are transformed into machine code through a compilation process.
  • Interpreted languages like Python and shell scripts are executed by an interpreter program, typically written in a compiled language, allowing these scripts to run as intended on the computer.
  • Programs consist of commands executed by computers.
  • Commands are in binary, interpreted differently by processor types.
  • High-level languages are either compiled or interpreted.
  • Compiled languages are transformed into machine code.
  • Interpreted languages run through an interpreter, enabling cross-platform compatibility.

Building Programs

Building a program involves transforming source code into an executable binary that can be run on a computer. This process typically involves several steps, such as preprocessing, compiling, linking, and packaging.

  • Preprocessing: The source code is processed to handle preprocessor directives, such as #include in C.
  • Compiling: The source code is translated into object files containing machine code.
  • Linking: The object files are combined to create an executable binary, resolving dependencies between them.
  • Packaging: The executable binary is bundled with any necessary resources, such as libraries and configuration files.
  • Building transforms source code into an executable binary.
  • Preprocessing handles directives like #include.
  • Compiling converts source to machine code in object files.
  • Linking merges object files into an executable, resolving dependencies.
  • Packaging includes necessary resources with the executable.

Build Systems

A build system is a tool that automates the construction of a program, using a defined set of tasks with precedence constraints. It computes a topological ordering of a graph to construct resources and manage dependencies.

  • Automates program construction to avoid repetitive tasks.
  • Manages dependencies to ensure consistent builds.
  • Uses a graph to determine the order of task execution.
  • Crucial for handling complex projects with many components.

Understanding Build Systems

Build systems ensure proper construction of all nodes in a dependency graph, automating updates and rebuilds when necessary.

  • Central to managing project dependencies.
  • Automatically rebuilds parts of a project when dependencies change.
  • Ensures that all project components are up to date.
  • Crucial for maintaining project efficiency and consistency.

Example: Factorial Program

Consider a simple program that calculates factorials. The program consists of two source files: fact.c and main.c. The fact.c file contains the factorial calculation function, while main.c contains the main function that calls the factorial function.

// fact.h
#ifndef FACT_H
#define FACT_H

long long fact(int);

#endif


// fact.c
#include "fact.h"
int factorial(int n) {
    if (n == 0) {
        return 1;
    } else {
        return n * factorial(n - 1);
    }
}


// main.c
#include <stdio.h>
#include <stdlib.h>
#include "fact.h"

int main(int argc, char *argv[])
{
  if (argc == 1) {
    printf("usage: %s n1 [n2 [...]]", argv[0]);
    printf("\n\nCompute the factorial of the provided numbers");
    return 0;
  }

  for (int i = 1; i < argc; i++) {
    int n = atoi(argv[i]);
    printf("%lld ", fact(n));
  }
  printf("\n");
}
  • Illustrates a multi-file program with fact.c for factorial calculation and main.c for program entry.
  • Demonstrates dependency management: main.c depends on fact.c.
  • Shows basic programming concepts: functions, loops, conditionals, and I/O.
  • Emphasizes the importance of header files for sharing function declarations between source files.

Example: Compiling the Program

To compile a simple C program that calculates factorials, we follow these steps:

gcc -c fact.c
gcc -c main.c
gcc -o main main.o fact.o

Updating any file requires re-running relevant commands.

  • Compiling a C program involves transforming source code (fact.c and main.c) into machine code (object files) and then linking these object files into an executable (main).
  • The command gcc -c compiles source files into object files separately.
  • The command gcc -o links object files into a single executable.
  • Changes in source files require recompilation and possibly re-linking to update the executable.
  • This process illustrates the linear dependency chain: source code -> object files -> executable.

  • This approach promotes reusability and modularity in build processes.

From Scripts to Structured Buildfiles

  • Evolution: From simple scripts to structured buildfiles.
  • Benefits: Improved modularity, flexibility, and complex dependency management.
  • Transition: Enables more efficient task execution.
  • The shift from scripts to buildfiles marks a significant improvement in how build processes are defined and managed.

  • This ANT buildfile example showcases basic but essential features: project structure, property definition, and task declaration.

Buildfile and Build Script Comparison

  • Buildfiles: Structured and declarative, with better control over dependency management and task execution.
  • Build Scripts: More direct, executing equivalent operations in a more imperative style.
  • Buildfiles offer a high-level, declarative approach focusing on “what” needs to be built, improving readability and maintainability.
  • Scripts, being more imperative, focus on “how” tasks are executed, requiring more detailed procedural logic.
  • The comparison underscores the efficiency and clarity benefits of buildfiles, particularly for managing complex or large-scale projects.

  • Parallelization: Difficulty in parallelizing build steps.
  • Incremental Builds: Challenges with managing incremental builds due to script unpredictability.

  • Parallelizing tasks is often more complicated, potentially leading to inefficiencies in build times and resource utilization.
  • Incremental builds, crucial for large projects, can be challenging due to the unpredictable nature of some tasks and dependencies.

Introducing Artifact-Based Build Systems

  • Benefits: More efficient, parallel, and incremental builds.

  • This approach allows for more granular dependency tracking, enabling efficient incremental builds and parallel execution.
  • The shift to artifact-centric builds addresses many of the limitations of task-based systems, offering scalability and performance improvements for modern software development.

The Functional Paradigm in Build Systems

  • Principles: Embody functional programming concepts.

Understanding Targets in Bazel

  • Targets: Defined in BUILD files, corresponding to artifacts.
  • Types: java_binary for executables, java_library for libraries.
  • Components:
    • name: How the target is referenced.
    • srcs: Source files to compile.
    • deps: Dependencies on other targets.
  • In Bazel, targets are the basic units of build, corresponding to outputs like binaries or libraries.
  • java_binary and java_library are examples of target types, indicating the nature of the artifact being built.
  • Each target has a name for identification, srcs for source files involved in the build, and deps for dependencies on other targets, ensuring all necessary components are included and up-to-date.

Dependency Management in Bazel

Dependencies in Bazel can be:

  • Within the same package (e.g., ":mylib").
  • In a different package within the source hierarchy (e.g., "//java/com/example/common").
  • Third-party artifacts outside the source hierarchy (e.g., "@com_google_common_guava_guava//jar").
  • Bazel’s dependency management system allows specifying dependencies within the same package, across packages within a project, and on third-party artifacts.
  • This flexibility ensures that all dependencies, whether internal or external, are correctly managed and resolved during the build process.
  • By managing dependencies explicitly, Bazel reduces build errors and ensures consistency across builds.

The Bazel Workspace

A Bazel workspace is identified by a WORKSPACE file at the root, organizing the source hierarchy for a project and defining external dependencies.

  • The WORKSPACE file marks the root of a Bazel project, serving as the entry point for the build system.
  • It organizes the project’s source hierarchy and specifies external dependencies, allowing Bazel to manage and fetch them as needed.
  • This file plays a crucial role in setting up a Bazel project, defining the workspace’s boundaries and external interfaces.

Building with Bazel

To build a target, such as MyBinary, execute:

bazel build :MyBinary

Bazel performs these steps:

  1. Parses BUILD files to create a dependency graph.
  2. Determines transitive dependencies of the target.
  3. Builds each dependency in order, then the target itself.
  • Step 1: Bazel scans BUILD files in the workspace to understand the structure of the project and how different parts depend on each other.
  • Step 2: It identifies all dependencies for the specified target, ensuring all needed components are accounted for before the build starts.
  • Step 3: Dependencies are built in a sequence that respects their interdependencies, culminating in the build of the specified target.
  • This methodical approach minimizes unnecessary rebuilds and ensures that any changes in dependencies trigger a rebuild of affected targets, enhancing efficiency.

Distributed Builds: The Frontier of Scalability

The combination of remote caching and remote execution enables distributed builds, allowing build processes to scale across multiple machines and significantly reducing build times.

  • Remote Caching: Stores build outputs in a shared cache, making it possible for subsequent builds to reuse previously computed results, dramatically speeding up the build process.
  • Remote Execution: Allows parts of the build process to be executed on remote machines, distributing the workload and further reducing build times.
  • Distributed builds are especially beneficial for large projects with complex dependencies, as they can significantly reduce the time required to compile and link large codebases.
  • This scalability is crucial for organizations with large teams and continuous integration/continuous deployment (CI/CD) pipelines, where build times can be a bottleneck.

Embracing Build Systems in Project Lifecycle

Choosing the right build system, whether task-based or artifact-based, depends on project scale and complexity, with modern artifact-based systems offering significant advantages for large-scale projects.

  • Project Scale and Complexity: The choice of build system should consider the size of the project and the complexity of its dependencies.
  • Task-based vs. Artifact-based: Task-based systems are suitable for simpler projects, but artifact-based systems, like Bazel, provide better scalability and efficiency for larger projects.
  • Advantages of Artifact-based Systems: These systems improve build times through parallelism, accurate dependency tracking, and caching, making them ideal for projects with heavy build requirements.

Dealing with Modules and Dependencies

Structuring Projects in Artifact-Based Systems

Projects using systems like Bazel are organized into modules defined in BUILD files, influencing build performance and maintenance efforts significantly.

  • Artifact-based build systems organize projects into modules.
  • Each module is defined within its own BUILD file.
  • The structure of these modules significantly affects both build performance and the ease of maintenance.

The 1:1:1 Rule for Fine-Grained Modules

Optimal module granularity involves balancing between a single module for the entire project and one-module-per-file, with a preference for smaller, manageable modules for improved build flexibility and efficiency.

  • Module granularity refers to the size and scope of modules within a project.
  • The 1:1:1 rule suggests a balanced approach, avoiding both monolithic and overly fragmented module structures.
  • Smaller, well-defined modules offer better build flexibility and efficiency.

Minimizing Module Visibility for Better Control

Module visibility should be minimized to what’s necessary, with public, private, or restricted visibility settings controlling access and ensuring encapsulation.

  • Module visibility settings control which other modules can depend on a given module.
  • Minimizing visibility helps enforce encapsulation and manage dependencies more securely.
  • Different visibility levels include public, private, and restricted.

Managing Internal Dependencies

In large projects, most dependencies are internal, built from the same source repository, highlighting the importance of explicit dependency management for build consistency.

  • Internal dependencies are those within the same project or source repository.
  • Clear and explicit management of these dependencies is crucial for ensuring build consistency.
  • Proper management aids in avoiding build conflicts and redundancy.

Handling Transitive Dependencies

Strict transitive dependency modes require targets to explicitly list all dependencies, preventing implicit reliance on transitive dependencies and improving build performance.

  • Transitive dependencies are indirect dependencies of a module.
  • Requiring explicit listing of all dependencies ensures that no unintended dependencies are included, improving build performance.
  • This practice also aids in better understanding and managing project dependencies.

Aside: Transitive Dependencies

Transitive dependencies refer to the indirect dependencies within a project’s module. These are the libraries or modules that your project’s direct dependencies rely on. Managing them explicitly is crucial to prevent build system from inadvertently relying on them, ensuring a more stable and predictable build process.

  • Transitive dependencies are not directly referenced by your project but are required by your project’s direct dependencies.
  • Overlooking transitive dependencies can lead to unexpected behavior or build failures, as the build system may assume the presence of these indirect dependencies.
  • By explicitly listing all dependencies, including transitive ones, you gain better control over your project’s dependency graph, reducing the risk of incompatibilities and enhancing build reproducibility.

Example:

Imagine your project, Project A, depends directly on Library B. Library B, in turn, requires Library C to function properly. Library C is a transitive dependency of your project. If Library B’s version is updated and it no longer requires Library C, but your project inadvertently relies on functionality from Library C, your project will break unless you have explicitly defined a dependency on Library C.

External Dependency Management

External dependencies, those built outside the project’s build system, should be managed manually to control versions and maintain build stability.

  • External dependencies come from outside the project’s own source repository.
  • Manual management of these dependencies is recommended to precisely control versions and ensure stability.
  • Careful version management prevents compatibility issues and maintains build reliability.

The One-Version Rule

To avoid the diamond dependency problem, projects should adhere to the One-Version Rule, enforcing a single version of any external library throughout the repository.

  • The diamond dependency problem occurs when multiple versions of a library are included in a project.
  • The One-Version Rule mandates a single version of any external library to be used across the entire project.
  • Adhering to this rule prevents conflicts and simplifies dependency management.

Challenges with Transitive External Dependencies

Managing the transitive dependencies of external libraries requires careful consideration to avoid version conflicts and ensure consistent, reliable builds.

  • Transitive external dependencies can introduce version conflicts and instability.
  • Careful consideration and management strategies are needed to handle these dependencies effectively.
  • Approaches like using a global manifest can help maintain consistency and avoid conflicts.

Caching and Security with External Dependencies

Using remote caching and mirroring for external dependencies can optimize build times while mitigating security and reliability risks.

  • Remote caching speeds up builds by reusing previously downloaded dependencies.
  • Mirroring external dependencies locally can enhance security and reliability by reducing reliance on third-party availability.
  • Both strategies contribute to faster, more secure builds.

Vendoring: An Alternative Approach

Vendoring, or checking dependencies into source control, transforms external dependencies into internal ones, sidestepping common issues with third-party sources.

  • Vendoring involves including external dependencies directly in a project’s source control.
  • This practice transforms external dependencies into internal ones, easing management and avoiding third-party issues.
  • While offering several benefits, vendoring must be managed carefully to avoid bloating the repository and complicating updates.
