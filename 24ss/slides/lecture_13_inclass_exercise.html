SOLID Principles in Python

Exploring SOLID Principles Through Practice

This exercise is based on the Real Python article titled “SOLID Design Principles in Python”.


In this exercise we will use example code to illustrate both violations and adherence to the SOLID principles within Python programming. By examining these principles in action, we aim to gain a deeper understanding of how they can be applied to real-world Python code, enhancing its design, maintainability, and scalability.

Single Responsibility Principle

SRP Main Idea

The single-responsibility principle states that:

A class should have only one reason to change.

The Single Responsibility Principle (SRP) advises that a class should be focused on a single aspect of the system’s functionality, encapsulating all the data and behaviors relevant to that aspect. By adhering to SRP, developers ensure that each class addresses a specific concern or functionality, making the system easier to understand, debug, and test. This principle plays a vital role in preventing classes from becoming overly complex and hard to maintain by limiting their responsibilities. Moreover, SRP facilitates easier modification and extension of the system since changes to one aspect of the system’s functionality are localized to a single class or module. This isolation of responsibilities enhances the system’s adaptability to new requirements or changes in the external environment without necessitating widespread modifications, thereby significantly reducing the risk of introducing errors into unrelated parts of the system. In essence, SRP contributes to the creation of a modular architecture where components can be independently developed, tested, and updated, leading to higher quality software that is both robust and flexible.

SOLID Violation

from pathlib import Path
from zipfile import ZipFile

class FileManager:
    def __init__(self, filename):
        self.path = Path(filename)

    def read(self, encoding="utf-8"):
        return self.path.read_text(encoding)

    def write(self, data, encoding="utf-8"):
        self.path.write_text(data, encoding)

    def compress(self):
        with ZipFile(self.path.with_suffix(".zip"), mode="w") as archive:
            archive.write(self.path)

    def decompress(self):
        with ZipFile(self.path.with_suffix(".zip"), mode="r") as archive:
            archive.extractall()

This class has two distinct sets of responsibilities: file management (reading and writing files) and compression management (compressing and decompressing files). Each set of functionalities operates independently within the class and serves different purposes. Therefore, changes in the way the class handles file operations (such as modifying the encoding method) or changes in the compression algorithm would require modifications to the class’s internal code. This dual focus means the class has more than one reason to change, which is against the Single Responsibility Principle (SRP). SRP advocates for a class to have only one reason to change, suggesting that FileManager should be refactored into separate classes that handle file management and compression/decompression tasks individually, thereby adhering to SRP and simplifying the internal implementation.

SRP Adherence

# file_manager_srp.py

from pathlib import Path
from zipfile import ZipFile

class FileManager:
    def __init__(self, filename):
        self.path = Path(filename)

    def read(self, encoding="utf-8"):
        return self.path.read_text(encoding)

    def write(self, data, encoding="utf-8"):
        self.path.write_text(data, encoding)

class ZipFileManager:
    def __init__(self, filename):
        self.path = Path(filename)

    def compress(self):
        with ZipFile(self.path.with_suffix(".zip"), mode="w") as archive:
            archive.write(self.path)

    def decompress(self):
        with ZipFile(self.path.with_suffix(".zip"), mode="r") as archive:
            archive.extractall()

In this refined example, we demonstrate adherence to the Single Responsibility Principle (SRP) by dividing the original FileManager class into two focused classes: FileManager and ZipFileManager. This division ensures that each class is dedicated to a single aspect of file management—FileManager to reading and writing operations, and ZipFileManager to handling ZIP file compression and decompression. This separation enhances clarity, maintainability, and scalability of the code.

A key takeaway here is the nuanced understanding of “responsibility.” It’s not about limiting classes to a single action or method but ensuring they address a singular, cohesive functional area. This approach simplifies debugging, testing, and future modifications, as changes are isolated to the specific class that encapsulates the relevant functionality. The division into more specialized classes reflects a commitment to SRP, showcasing a practical application of this principle to improve software design and architecture.

Open/Closed Principle

OCP Main Idea

The open/closed principle states that:

You should be able to extend a class’s behavior without modifying it.

The Open/Closed Principle (OCP), one of the SOLID principles introduced by Robert C. Martin, emphasizes that software entities (such as classes, modules, functions, etc.) should be open for extension but closed for modification. This principle aims to promote software designs that are resilient to change and easy to update or extend without the need to alter existing source code. Adherence to OCP reduces the risk of introducing bugs when new features are added or when the system is extended in other ways. It encourages developers to use interfaces and abstract classes to define stable, flexible abstractions that can be extended with new behaviors through subclassing or implementing these interfaces, thereby achieving extensibility while preserving the integrity of the original codebase. This design approach significantly enhances the scalability and robustness of software systems, making them more adaptable to changing requirements over time.

SOLID Violation

# shapes_ocp.py

from math import pi

class Shape:
    def __init__(self, shape_type, **kwargs):
        self.shape_type = shape_type
        if self.shape_type == "rectangle":
            self.width = kwargs["width"]
            self.height = kwargs["height"]
        elif self.shape_type == "circle":
            self.radius = kwargs["radius"]

    def calculate_area(self):
        if self.shape_type == "rectangle":
            return self.width * self.height
        elif self.shape_type == "circle":
            return pi * self.radius**2

This Python code illustrates a violation of the Open/Closed Principle (OCP), which states that software entities (classes, modules, functions, etc.) should be open for extension, but closed for modification. The Shape class is designed to calculate the area of shapes but does so in a way that is not conducive to extension. The calculate_area method uses type checking (if statements based on shape_type) to determine which calculation to perform. This approach requires modifying the Shape class whenever a new shape needs to be supported, going against the “closed for modification” aspect of OCP. To add support for a new shape, the developer must alter the existing code of the Shape class, adding another elif condition for the new shape. This not only increases the risk of introducing bugs into the existing code but also makes the class more complex and harder to maintain over time.

OCP Adherence

# shapes_ocp.py

from abc import ABC, abstractmethod
from math import pi

class Shape(ABC):
    def __init__(self, shape_type):
        self.shape_type = shape_type

    @abstractmethod
    def calculate_area(self):
        pass

class Circle(Shape):
    def __init__(self, radius):
        super().__init__("circle")
        self.radius = radius

    def calculate_area(self):
        return pi * self.radius**2

class Rectangle(Shape):
    def __init__(self, width, height):
        super().__init__("rectangle")
        self.width = width
        self.height = height

    def calculate_area(self):
        return self.width * self.height

class Square(Shape):
    def __init__(self, side):
        super().__init__("square")
        self.side = side

    def calculate_area(self):
        return self.side**2

This updated code exemplifies adherence to the Open/Closed Principle (OCP) by restructuring the initial Shape class into an abstract base class (ABC) and defining separate classes for each shape that inherit from this base class. The base class Shape declares an abstract method calculate_area, which is implemented by the derived classes Circle, Rectangle, and Square. This design allows for the addition of new shapes without modifying the existing codebase, as each new shape can be represented by a new derived class that implements the calculate_area method. This approach ensures that the existing code is closed for modification while remaining open for extension, thereby adhering to the OCP and promoting a more scalable and maintainable software architecture.

Liskov Substitution Principle

LSP Main Idea

The Liskov substitution principle states that:

Subtypes must be substitutable for their base types.

The Liskov Substitution Principle (LSP) asserts that objects of a superclass should be replaceable with objects of its subclasses without affecting the correctness of the program. This principle is vital for maintaining the integrity of the software design, as it guarantees that subclasses extend the base classes without changing their behavior in unexpected ways. Adherence to LSP ensures that a system is designed with a proper hierarchy where derived classes truly represent “is-a” relationships with their base classes, promoting reusability and robustness. LSP is crucial for designing systems that are easy to understand, maintain, and scale, as it enforces consistency and predictability across the inheritance hierarchy, allowing developers to use polymorphism effectively.

SOLID Violation

# shapes_lsp.py

class Rectangle:
    def __init__(self, width, height):
        self.width = width
        self.height = height

    def calculate_area(self):
        return self.width * self.height


class Square(Rectangle):
    def __init__(self, side):
        super().__init__(side, side)

    def __setattr__(self, key, value):
        super().__setattr__(key, value)
        if key in ("width", "height"):
            self.__dict__["width"] = value
            self.__dict__["height"] = value

This example highlights a violation of the Liskov Substitution Principle (LSP) using a Rectangle class and its subclass Square. According to LSP, if class B is a subclass of class A, instances of A should be replaceable with instances of B without affecting the correctness of the program. However, this principle is breached in the given code.

The Rectangle class calculates area based on width and height, assuming these dimensions are independent. The Square class, derived from Rectangle, enforces equal width and height to maintain square properties, which introduces unexpected behavior when Square is used in place of Rectangle. The __setattr__ method in Square synchronizes width and height, violating the independence expected from a Rectangle.

The core issue is that while a square is mathematically a rectangle (all sides equal), the code design does not respect this relationship in a manner consistent with LSP. Substituting Rectangle with Square could lead to unpredictable outcomes, especially in scenarios where width and height are assumed to be independent. This design flaw undermines the polymorphic nature of the objects, making the Square class incompatible with the expected behavior of its superclass, Rectangle.

To comply with LSP, classes representing geometric shapes like rectangles and squares should be designed to ensure that subclasses can be used interchangeably with their parent classes without altering the program’s correctness. One solution could be to avoid inheritance between these two classes or to redesign the class hierarchy to better reflect the behaviors and properties of geometric shapes.

LSP Adherence

# shapes_lsp.py

from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def calculate_area(self):
        pass

class Rectangle(Shape):
    def __init__(self, width, height):
        self.width = width
        self.height = height

    def calculate_area(self):
        return self.width * self.height

class Square(Shape):
    def __init__(self, side):
        self.side = side

    def calculate_area(self):
        return self.side ** 2


>>> from shapes_lsp import Rectangle, Square
>>> def get_total_area(shapes):
...     return sum(shape.calculate_area() for shape in shapes)
>>> get_total_area([Rectangle(10, 5), Square(5)])
75

One way to solve this problem is to create a base class for both Rectangle and Square to extend. This base class, Shape, defines a method calculate_area that must be implemented by its subclasses. This ensures that all subclasses of Shape will have a consistent interface for calculating area, allowing them to be used interchangeably.

Shape becomes the type that you can substitute through polymorphism with either Rectangle or Square, which are now siblings rather than a parent and a child. Notice that both concrete shape types have distinct sets of attributes, different initializer methods, and could potentially implement even more separate behaviors. The only thing that they have in common is the ability to calculate their area.

With this implementation in place, you can use the Shape type interchangeably with its Square and Rectangle subtypes when you only care about their common behavior.

Interface Segregation Principle

ISP Main Idea

The interface segregation principle states that:

Clients should not be forced to depend upon methods that they do not use. Interfaces belong to clients, not to hierarchies.

The Interface Segregation Principle (ISP) is a fundamental design guideline in object-oriented programming that emphasizes the importance of creating interfaces that are client-specific rather than general-purpose. This principle stems from the realization that no client should be forced to depend on methods it does not use, which can lead to bloated, cumbersome interfaces that are difficult to implement and maintain. By segregating interfaces according to the specific needs of client modules, ISP ensures that a system remains flexible and that its components can be easily adapted or replaced without impacting clients that do not use those interfaces. Essentially, ISP encourages the design of modular, decoupled systems where each component communicates through well-defined, minimal interfaces, thereby enhancing code maintainability, scalability, and robustness.

SOLID Violation

from abc import ABC, abstractmethod

class Printer(ABC):
    @abstractmethod
    def print(self, document):
        pass

    @abstractmethod
    def fax(self, document):
        pass

    @abstractmethod
    def scan(self, document):
        pass

class OldPrinter(Printer):
    def print(self, document):
        print(f"Printing {document} in black and white...")

    def fax(self, document):
        raise NotImplementedError("Fax functionality not supported")

    def scan(self, document):
        raise NotImplementedError("Scan functionality not supported")

class ModernPrinter(Printer):
    def print(self, document):
        print(f"Printing {document} in color...")

    def fax(self, document):
        print(f"Faxing {document}...")

    def scan(self, document):
        print(f"Scanning {document}...")

Consider the following example of class hierarchy to model printing machines.

In this example, the base class, Printer, provides the interface that its subclasses must implement. OldPrinter inherits from Printer and must implement the same interface. However, OldPrinter doesn’t use the .fax() and .scan() methods because this type of printer doesn’t support these functionalities.

This implementation violates the ISP because it forces OldPrinter to expose an interface that the class doesn’t implement or need.

ISP Adherence

# printers_isp.py

from abc import ABC, abstractmethod

class Printer(ABC):
    @abstractmethod
    def print(self, document):
        pass

class Fax(ABC):
    @abstractmethod
    def fax(self, document):
        pass

class Scanner(ABC):
    @abstractmethod
    def scan(self, document):
        pass

class OldPrinter(Printer):
    def print(self, document):
        print(f"Printing {document} in black and white...")

class NewPrinter(Printer, Fax, Scanner):
    def print(self, document):
        print(f"Printing {document} in color...")

    def fax(self, document):
        print(f"Faxing {document}...")

    def scan(self, document):
        print(f"Scanning {document}...")

To fix this issue, you should separate the interfaces into smaller and more specific classes. Then you can create concrete classes by inheriting from multiple interface classes as needed. Now Printer, Fax, and Scanner are base classes that provide specific interfaces with a single responsibility each. To create OldPrinter, you only inherit the Printer interface. This way, the class won’t have unused methods. To create the ModernPrinter class, you need to inherit from all the interfaces. In short, you’ve segregated the Printer interface.

This class design allows you to create different machines with different sets of functionalities, making your design more flexible and extensible.

Dependency Inversion Principle

DIP Main Idea

The dependency inversion principle states that:

Abstractions should not depend upon details. Details should depend upon abstractions.

The Dependency Inversion Principle (DIP) is crucial for achieving a loosely coupled architecture in software development. By advocating for high-level modules (those with complex logic) and low-level modules (those with detailed implementations) to both rely on abstractions, DIP ensures that changes to the detailed implementations of a system do not directly impact its high-level logic. This approach facilitates easier maintenance and scalability, as it reduces the direct dependencies between different parts of the code. Essentially, DIP allows for higher-level components to define their own abstractions, and for lower-level components to implement these abstractions, thereby inverting the traditional dependency relationship. This strategy promotes a more modular and cohesive code structure, where components can be easily modified or replaced without affecting the overall system functionality.

DIP Violation

    def display_data(self):

class BackEnd:
    def get_data_from_database(self):
        return "Data from the database"

In this example, the FrontEnd class depends on the BackEnd class and its concrete implementation. You can say that both classes are tightly coupled. This coupling can lead to scalability issues. For example, say that your app is growing fast, and you want the app to be able to read data from a REST API. How would you do that?

You may think of adding a new method to BackEnd to retrieve the data from the REST API. However, that will also require you to modify FrontEnd, which should be closed to modification, according to the open-closed principle.

DIP Adherence

from abc import ABC, abstractmethod

class FrontEnd:
    def __init__(self, data_source):
        self.data_source = data_source

    def display_data(self):
        data = self.data_source.get_data()
        print("Display data:", data)

class DataSource(ABC):
    @abstractmethod
    def get_data(self):
        pass

class Database(DataSource):
    def get_data(self):
        return "Data from the database"

class API(DataSource):
    def get_data(self):
        return "Data from the API"


>>> from app_dip import API, Database, FrontEnd

>>> db_front_end = FrontEnd(Database())
>>> db_front_end.display_data()
Display data: Data from the database

>>> api_front_end = FrontEnd(API())
>>> api_front_end.display_data()
Display data: Data from the API

To fix the issue, you can apply the dependency inversion principle and make your classes depend on abstractions rather than on concrete implementations like BackEnd. In this specific example, you can introduce a DataSource class that provides the interface to use in your concrete classes.

In this redesign of your classes, you’ve added a DataSource class as an abstraction that provides the required interface, or the .get_data() method. Note how FrontEnd now depends on the interface provided by DataSource, which is an abstraction.

Then you define the Database class, which is a concrete implementation for those cases where you want to retrieve the data from your database. This class depends on the DataSource abstraction through inheritance. Finally, you define the API class to support retrieving the data from the REST API. This class also depends on the DataSource abstraction.

Here, you first initialize FrontEnd using a Database object and then again using an API object. Every time you call .display_data(), the result will depend on the concrete data source that you use. Note that you can also change the data source dynamically by reassigning the .data_source attribute in your FrontEnd instance.
